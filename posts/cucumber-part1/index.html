cucumber set up

Part 1 of a few on Cucumber - setting it up with JUnit5 & Maven.
February 28, 2021 · 5 min

I recently wrote an integration test framework for a Kafka pipeline using Cucumber and Testcontainers. This is part 1 of a series explaining how, starting with setting up our project to be able to run Cucumber tests from maven & our IDE (IntelliJ in my case).

why cucumber?#

I’d previously written a couple of integration test frameworks triggered from JUnit @Test methods. These both used fluent APIs to set up the required environment & execute the test, hiding the plumbing from the test author.

This worked well in a team where the developers wrote most of the tests. This time around and on a different project I wanted tests that could be maintained by less-technical folks and hopefully replace separate acceptance criteria in stories.
I looked at a couple of BDD frameworks to achieve this and liked the look of Cucumber.

features and steps#

The key technical differences between writing a JUnit test and a Cucumber test (one implemented in Java at least) are that the tests are defined in feature files and use steps which are defined in a class file.

The feature file is written in Gherkin, a business-readable language for defining behaviour. It has a few keywords used to define the preconditions, stimulus & expected outcome of the test in a Given-When-Then format.

The example used in the 10min Cucumber tutorial looks a bit like this:

Feature: Is it Friday yet?
  Everybody wants to know when it's Friday

  Scenario: Sunday isn't Friday
    Given today is Sunday
    When I ask whether it's Friday yet
    Then I should be told "Nope"

… and the corresponding step definitions look like this (where IsItFriday is the ‘system’ under test):

public class StepDefinitions {
    private String today;
    private String actualAnswer;

    @Given("today is Sunday")
    public void todayIsSunday() {
        today = "Sunday";
    }

    @When("I ask whether it's Friday yet")
    public void iAskWhetherItIsFridayYet() {
        actualAnswer = IsItFriday.isItFriday(today);
    }

    @Then("I should be told {string}")
    public void iShouldBeTold(String expectedAnswer) {
        assertEquals(expectedAnswer, actualAnswer);
    }
}

(I am showing annotation-based step definitions here as I personally find them more readable than the lambda alternatives available as part of cucumber-java8).

system under test (SUT)#

My real SUT involved multiple kafka-steams components and a number of topics but here we’ll use a simplified example with a single kafka-stream that will receive details of orders that have been placed from one topic, enrich them with details of the (existing) customer that placed the order and provide a summary of the order to the sales team via another topic.
All messages will be in json.

project set-up#

The build-tool support for using Cucumber with JUnit 5 still has some gaps compared to the support for JUnit 4 (mostly around test discovery in Surefire) but I didn’t want to start a new project using an older version of JUnit so we’ll use JUnit 5 here.

Our initial project structure will look like this:

pom.xml
src
    test
        java
            dev.codecrumbs.sales.test.steps
                StepDefinitions.java
            features
                IntegrationTests.java                
        resources
            cucumber.properties
            junit-platform.properties
            features
                sales-ingestion.feature

A few things to note:

  • In the pom, we depend on cucumber-junit-platform-engine & cucumber-java. The former allows us to run Cucumber tests using the JUnit platform1 & the latter contains the annotations used to define test steps.
  • For feature files to be found (by surefire at least), it is necessary to have a class annotated with @Cucumber in the same package. In our case this is named IntegrationTests.
  • cucumber.properties contains settings that customise Cucumber (just output formatting for now) and is used by the Cucumber CLI when executing tests from your IDE.
  • junit-platform.properties is identical to cucumber.properties but is used by surefire. This duplication is an annoyance I have yet to resolve.

first feature#

sales-ingestion.feature looks like this:

Feature: Sales are ingested.
  Scenario: Details of a sale are ingested.
    Scenario Outline:
      Given An existing customer "<customer>" exists.
      When Order "<order>" is placed.
      Then The Sales Team are notified of Order "<summary>".
    Scenarios:
      | customer   | order       | summary            |
      | john-smith | five-apples | john-smiths-apples |

We have a single parameterized scenario that allows us to execute the same test multiple times with different inputs & expected outputs. These appear in the Scenarios table. As our inputs & expected outputs will be json, I didn’t want to clutter my features with inline json so intend to have customer, order and summary refer to external files.

the step definitions#

public class StepDefinitions {
    private static final Logger LOG = 
        LoggerFactory.getLogger(StepDefinitions.class);

    @Given("An existing customer {string} exists.")
    public void anExistingCustomer(String existingCustomer) {
        LOG.info("Loading existing customer {}", existingCustomer);
    }

    @When("Order {string} is placed.")
    public void orderIsReceived(String order) {
        LOG.info("Injecting order {}", order);
    }

    @Then("The Sales Team are notified of Order {string}.")
    public void salesTeamAreNotifiedOfOrder(String orderSummary) {
        LOG.info(
          "Verifying that order summary {} is received by sales team.", 
          orderSummary
        );
    }
}

These step definitions will interact with the system-under test but, for now, we are just logging that the step was executed.
